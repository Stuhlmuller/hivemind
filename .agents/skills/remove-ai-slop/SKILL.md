# Remove AI Slop

## Goal

Make Hivemind feel built by a careful security engineer, not generated by a generic assistant. Remove fluff, weak claims, decorative filler, overbroad abstractions, fake completeness, and UI or docs that describe more than the code actually does.

## Workflow

1. Inspect the changed files before editing.
2. Identify the exact slop category: vague language, unearned security claim, generic UI, overbuilt code, hidden credential risk, missing validation, weak test, or theme overuse.
3. Replace it with concrete behavior, narrower copy, simpler code, or a test-backed implementation.
4. Run the smallest meaningful verification.
5. Summarize what was tightened and what risk remains.

## Hivemind Standards

- Preserve the core security boundary: agents request brokered capabilities; agents never receive raw credentials.
- Treat every credential statement as a claim that must be true in code, config, or tests.
- Prefer just-in-time, short-lived, least-privilege language over broad promises like "secure", "safe", or "enterprise-grade".
- Keep the bee theme functional: swarm, hive, scout, worker, forager, lease, broker, intent. Avoid cute filler that makes security behavior less clear.
- Make agent communication short, actionable, and low-context.
- Prefer boring, inspectable code over clever frameworks until the repo earns more machinery.
- Keep the single-container promise intact unless the user explicitly changes the architecture.
- Commit often when doing multi-step work: small checkpoints after meaningful features, fixes, or security boundary changes.

## Slop To Remove

### Copy And Docs

Replace:

- "robust", "seamless", "powerful", "secure by design", "military grade", "production ready", "AI-powered" without specifics.
- Long paragraphs that restate the same idea.
- Product copy that promises integrations or policies not implemented.
- Explanations of obvious UI controls.

Use:

- Direct statements of current behavior.
- Explicit TODO or "next step" wording when something is planned but not implemented.
- API names, config keys, TTLs, policy fields, and test names when helpful.

### Code

Remove:

- Abstractions with only one caller and no clear security payoff.
- Placeholder providers that look operational but silently do nothing.
- Policy bypasses, raw token returns, broad exception swallowing, and mutable global secrets.
- Stringly typed security decisions when a small enum, dataclass, or validator is clearer.

Prefer:

- Narrow services with explicit inputs and outputs.
- Deny-by-default policy paths.
- Public views that redact or omit secrets.
- Tests that prove denial, expiry, scoping, and audit behavior.

### Frontend

Remove:

- Marketing landing pages when the user asked for an app.
- Decorative-only bee elements that crowd the workflow.
- Huge hero text inside operational panels.
- UI text that explains how to use basic controls.
- One-note amber/yellow palettes.

Prefer:

- A usable first screen with agents, credentials, leases, actions, and audit state.
- Compact operational panels.
- Visible status, errors, and audit outcomes.
- Responsive layouts that do not hide form controls or overflow buttons.

## Review Checklist

Before finishing a de-slop pass, check:

- Does each security claim correspond to implemented behavior or an explicit future note?
- Can an agent obtain a raw credential anywhere?
- Are lease TTL, action scope, and policy denial paths tested?
- Is the UI doing real work rather than narrating capabilities?
- Does the README tell users how to run the current app, not an imagined app?
- Did the change add dependency, config, or architecture weight without a concrete need?
- Did the final answer avoid hype and clearly state verification?
